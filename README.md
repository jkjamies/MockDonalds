# MockDonalds

A Kotlin Multiplatform (KMP) demo app showcasing shared business logic with native UI on both platforms: Jetpack Compose on Android and SwiftUI on iOS.

## Architecture

```
Kotlin commonMain (shared)                Kotlin androidMain
──────────────────────────                ──────────────────
Screen definitions                        Compose UI (@CircuitInject)
Presenter + UiState/Events                Ui.Factory (generated by circuit-codegen)
Presenter.Factory (generated)
Metro DI graph (AppGraph)

Kotlin iosMain (bridge)                   Swift (iOS native)
───────────────────────                   ──────────────────
IosApp (Metro graph + Circuit)            CircuitIos (presenter + UI resolution)
CircuitPresenterKotlinBridge              CircuitView / CircuitContent / CircuitStack
  (Molecule + @NativeCoroutinesState)     ScreenUiFactory<Screen, State> { View }
                                          Per-screen SwiftUI Views
```

**Shared via Circuit**: Screen, Presenter, UiState, Events, Navigator, Presenter.Factory, Metro DI

**Platform-specific**: UI rendering only (Compose on Android, SwiftUI on iOS)

### Key Libraries

| Library | Role |
|---------|------|
| [Circuit](https://slackhq.github.io/circuit/) | Presenter/UI/Screen pattern, navigation |
| [Metro](https://zacsweers.github.io/metro/) | Dependency injection (KMP, compile-time) |
| [Molecule](https://github.com/cashapp/molecule) | Bridges `@Composable` presenters to `StateFlow` for iOS |
| [KMP-NativeCoroutines](https://github.com/rickclephas/KMP-NativeCoroutines) | Bridges Kotlin `StateFlow` to Swift `AsyncSequence` |

### Data Flow

```
Android:
  Presenter.present() ──[Compose Runtime]──> Compose UI

iOS:
  Presenter.present() ──[Molecule]──> StateFlow ──[KMP-NativeCoroutines]──> AsyncSequence ──> SwiftUI
```

## Project Structure

```
features/
  {feature}/
    api/          # Screen definition, navigation contract
    data/         # Repository implementations
    domain/       # Use cases, domain models
    presentation/
      src/commonMain/   # Presenter, UiState, Events (shared)
      src/androidMain/  # Compose UI (Android only)

composeApp/
  src/commonMain/       # AppGraph (Metro DI), app-level wiring
  src/iosMain/          # IosApp, CircuitPresenterKotlinBridge

iosApp/
  iosApp/
    Circuit/            # CircuitIos, CircuitView, CircuitContent, CircuitStack
    Theme/              # MockDonaldsColors
    Features/{Feature}/ # SwiftUI Views
```

## How to Add a New Screen

### 1. Screen Definition (Kotlin commonMain)

Create the Screen in the feature's API module:

```
features/{feature}/api/src/commonMain/kotlin/.../navigation/{Feature}Screen.kt
```

```kotlin
@Parcelize
data object FeatureScreen : Screen
```

### 2. UiState & Events (Kotlin commonMain)

```
features/{feature}/presentation/src/commonMain/kotlin/.../{Feature}UiState.kt
```

```kotlin
data class FeatureUiState(
    // Add your state properties here
    val title: String,
    val items: List<Item>,
    val eventSink: (FeatureEvent) -> Unit,
) : CircuitUiState

sealed interface FeatureEvent {
    data class ItemClicked(val id: String) : FeatureEvent
}
```

### 3. Presenter (Kotlin commonMain)

```
features/{feature}/presentation/src/commonMain/kotlin/.../{Feature}Presenter.kt
```

```kotlin
@CircuitInject(FeatureScreen::class, AppScope::class)
@Composable
fun FeaturePresenter(
    navigator: Navigator,
): FeatureUiState {
    // Compose runtime — use remember, LaunchedEffect, etc.
    return FeatureUiState(
        title = "Hello",
        items = emptyList(),
        eventSink = { event ->
            when (event) {
                is FeatureEvent.ItemClicked -> navigator.goTo(DetailScreen(event.id))
            }
        },
    )
}
```

`@CircuitInject` + circuit-codegen generates the `Presenter.Factory` automatically. Metro discovers it via `@ContributesIntoSet`. No manual registration needed.

### 4. Android Compose UI (Kotlin androidMain)

```
features/{feature}/presentation/src/androidMain/kotlin/.../{Feature}Ui.kt
```

```kotlin
@CircuitInject(FeatureScreen::class, AppScope::class)
@Composable
fun FeatureUi(state: FeatureUiState, modifier: Modifier = Modifier) {
    // Jetpack Compose UI
    Column(modifier = modifier) {
        Text(state.title)
    }
}
```

Same `@CircuitInject` pattern generates the `Ui.Factory`. Only runs on Android via `kspAndroid`.

### 5. iOS SwiftUI View

```
iosApp/iosApp/Features/{Feature}/{Feature}View.swift
```

```swift
import SwiftUI
import ComposeApp

struct FeatureView: View {
    let state: FeatureUiState

    var body: some View {
        VStack {
            Text(state.title)
        }
    }
}
```

The view is a pure function of state — no observation logic, no presenters.

### 6. Register the iOS UI Factory

In `AppDelegate.swift`, add one line to the `uiFactories` array:

```swift
ScreenUiFactory<FeatureScreen, FeatureUiState> { FeatureView(state: $0) },
```

That's it. `ScreenUiFactory` is a generic that handles screen type matching and state casting.

### 7. Wire into Navigation

For tab-based screens, add to `MockDonaldsApp.swift`:

```swift
CircuitContent(screen: FeatureScreen.shared)
    .tabItem { Label("FEATURE", systemImage: "star") }
```

### 8. Export the Module (if new feature module)

In `composeApp/build.gradle.kts`:

```kotlin
// iOS framework block:
export(project(":features:{feature}:presentation"))

// commonMain.dependencies:
api(project(":features:{feature}:presentation"))
```

### Summary: What Goes Where

| What | Where | Why |
|------|-------|-----|
| Screen | `api/commonMain` | Shared navigation contract |
| Presenter | `presentation/commonMain` | Shared business logic, auto-discovered by Metro |
| UiState/Events | `presentation/commonMain` | Shared state contract |
| Compose UI | `presentation/androidMain` | Android-only, `@CircuitInject` generates Ui.Factory |
| SwiftUI View | `iosApp/Features/` | iOS-only, pure state function |
| UI registration | `AppDelegate.swift` | One-liner per screen via `ScreenUiFactory` |

### Convention Plugin

The `mockdonalds.kmp.compose` convention plugin handles all build configuration for presentation modules:

- Applies Compose, Metro, and KSP plugins
- Makes `circuit-runtime-presenter` an `api` dependency (so types appear in iOS framework headers)
- Runs circuit-codegen KSP per-target:
  - **Android**: sees presenters (commonMain) + UIs (androidMain) — generates both factory types
  - **iOS targets**: see presenters (commonMain) only — generates presenter factories only

### iOS Bridge Layer

The iOS bridge is a one-time setup in `composeApp/src/iosMain/`:

- **`IosApp`** — creates the Metro DI graph (`createGraph<AppGraph>()`), exposes `presenterBridge(screen:)` which resolves any screen's presenter via Circuit
- **`CircuitPresenterKotlinBridge`** — wraps a `@Composable Presenter.present()` call into a `StateFlow` via Molecule, annotated with `@NativeCoroutinesState` for Swift async observation

The Swift Circuit infrastructure (`CircuitIos`, `CircuitView`, `CircuitContent`, `CircuitStack`) consumes these bridges. `CircuitView` observes state via:

```swift
.task {
    let sequence = asyncSequence(for: presenter.stateFlow)
    for try await state in sequence { self.state = state }
}
```

Task cancellation is automatic when the view disappears.

## Building

```bash
# Android
./gradlew :composeApp:assembleDebug

# iOS framework
./gradlew :composeApp:linkDebugFrameworkIosSimulatorArm64

# iOS app — open in Xcode
open iosApp/iosApp.xcodeproj
```

The Xcode project includes a "Compile Kotlin Framework" build phase that runs the Gradle task automatically.
